# Alpha AI App 运维与 SLA 说明

## 当前结论

- 这套 App 现在已经具备基础可维护性:
  - 原生端和后端已拆分
  - 后端已有健康检查、metrics、运维摘要
  - App 内已有运维诊断页
- 这套 App 现在已经具备基础可监控性:
  - 活性: `/health/live`
  - 就绪: `/health/ready`
  - 指标: `/metrics`
  - 摘要: `/api/ops/summary`
- 这套 App **现在不能承诺 `99.99% SLA`**

原因很直接:

- 目前还是单机部署
- 核心状态里仍有 JSON / 本地文件依赖
- 没有反向代理层和双实例切换
- 没有外部 uptime 监控和告警升级
- 没有自动故障转移
- iOS/Android 包分发链还在内测收口阶段

## 现实可达目标

按当前代码形态:

- 本机/单机内测环境:
  - 可维护: `是`
  - 可监控: `是`
  - 可承诺 SLA: `不建议对外承诺`

按补最小运维底座后的单机生产:

- `systemd/supervisor + nginx + 健康探针 + 外部告警`
- 目标可先定 `99.5% 到 99.9%`

想逼近 `99.99%`:

- 至少要上双实例或多实例
- API 层前面加负载均衡/反向代理
- 状态存储迁到 `Postgres + Redis`
- 文件状态改数据库或对象存储
- 有自动重启、滚动发布、回滚
- 有外部监控、告警、值守
- 有压测与容量基线

## 已落地的监控入口

- API 活性:
  - `GET /health/live`
- API 就绪:
  - `GET /health/ready`
- Prometheus 文本指标:
  - `GET /metrics`
- 运维摘要:
  - `GET /api/ops/summary`
  - `GET /api/app/ops/summary`
- App 内入口:
  - “我的” -> “运维诊断”

## 当前可观测项

- 请求总量
- 5xx 错误量
- 错误率
- 平均延迟
- P95 延迟
- 最近错误时间与路径
- WebSocket 连接数
- 系统健康分
- 今日信号数
- 活跃策略数
- 数据源状态:
  - `scorecard`
  - `trade_journal`
  - `signals`
  - `positions`
  - `feedback`
  - `push devices`

## 交付建议

如果你准备小范围内测，建议按这个顺序上:

1. 先出 Android 预览包，内测 5 到 20 人。
2. 后端挂 `systemd` 或 `supervisor`，保证异常退出自动拉起。
3. 反代接 `nginx`，把 `/health/ready` 接进探针。
4. 外部监控至少补一层:
   - HTTP 可用性探测
   - 进程存活告警
   - 磁盘空间告警
5. 收第一轮反馈后，再决定是否投入 `99.9%` 以上的架构成本。

## 一句话结论

现在这版可以做小范围内测交付，也已经开始可维护、可监控；但如果你要的是严格意义上的 `99.99% SLA`，当前架构还不够，必须再上高可用基础设施。
